## DA System Debate 보강 리서치

### 질문 1 — Claude Haiku (또는 소형 LLM) JSON schema 준수율

*   **(a) 관련 공개 지표/연구**: Anthropic, OpenAI 등 주요 LLM 제공사의 tool-use/JSON mode 기능은 모델의 생성 과정을 제약(constrained decoding)하는 방식입니다. 따라서 공식 문서 외 별도 연구는 드물지만, 이 기능을 사용할 경우 구문(syntactic) 준수율은 사실상 100%에 가깝다고 보고됩니다. 단순 프롬프트 기반 요청 시에는 80-95%로 떨어질 수 있습니다.

*   **(b) 실무 권고치 범위**:
    *   API의 schema-enforced 기능 사용 시: **>99.5%**
    *   자유 문자열 출력 대비 enum 제한 출력의 위반율 감소 효과는 API 기능 사용 시 미미하나, 미사용 시에는 5-15%p 가량의 준수율 향상을 기대할 수 있습니다.
    *   준수율을 가장 크게 높이는 것은 프롬프트 엔지니어링보다 **API의 전용 JSON 출력 기능을 사용**하는 것입니다.

*   **(c) 우리 상황과의 주요 차이**:
    1.  우리의 schema (`{is_sufficient: bool, missing_da_ids: string[]}`)는 매우 단순하여, 복잡한 중첩 구조의 벤치마크보다 준수율이 높게 나올 가능성이 큽니다.
    2.  우리의 평가는 구문(syntactic) 준수를 넘어 의미(semantic)적 정확성(누락 ID를 정확히 찾아냈는가)까지 포함할 수 있습니다.
    3.  Haiku 모델 자체의 버전업에 따른 성능 변화가 공개 벤치마크 시점과 다를 수 있습니다.

### 질문 2 — Multilingual few-shot cap 실무 권고

*   **(a) 관련 공개 지표/연구**: "An Empirical Study of In-Context Learning" (Min et al., 2022) 등 다수 연구에서 예제의 수보다 **품질, 순서, 분포**가 더 중요하다고 보고합니다. GPT-3 논문(Brown et al., 2020)에서도 소수 예제로 큰 성능 향상을 보였습니다.

*   **(b) 실무 권고치 범위**:
    *   일반적인 few-shot cap: **3-5개**.
    *   Diminishing return 지점: 보통 **5-8개**를 넘어가면 성능 향상이 미미해지거나, 오히려 불필요한 정보로 인해 성능이 저하될 수 있습니���.
    *   특정 도메인 예제만 7건 주입 시, 해당 도메인 용어나 스타일에 과적합되어 다른 도메인 번역 요청에 대해 부자연스러운 결과를 생성할 위험이 큽니다.

*   **(c) 우리 상황과의 주요 차이**:
    1.  우리의 태스크는 일반 번역이 아닌 특정 'DA System' 도메인에 한정되어, 일반 벤치마크보다 적은 수의 고품질 예제가 더 효과적일 수 있습니다.
    2.  실시간 보조 태스크이므로, few-shot 예제 증가로 인한 latency 증가가 정확도 향상보다 더 민감한 제약 조건일 수 있습니다.
    3.  한국어-영어 쌍은 다른 언어 쌍에 비해 LLM이 비교적 강점을 보여 few-shot 의존도가 낮을 수 있습니다.

### 질문 3 — RAG 시스템 keyword dictionary 확장에서 uplift 관행

*   **(a) 관련 공개 지표/연구**: 이 주제는 상업적 RAG 시스템의 운영 노하우에 가까워 공개 연구가 매우 드뭅니다. 관련 학술 개념은 "Pseudo-Relevance Feedback"이지만, 보통 자동화된 확장을 다룹니다. 기업 기술 블로그에서 간헐적으로 아이디어를 얻을 수 있습니다.

*   **(b) 실무 권고치 범위**:
    *   Reported hit rate uplift: 초기 시스템의 경우 **+5%p ~ +20%p**까지 다양하나, 성숙한 시스템에서는 +1-5%p도 유의미한 향상으로 봅니다.
    *   Sunset clause는 업계 표준 관행입니다. 보통 "추가 후 **1-3개월**간 주요 지표(hit rate, user feedback 등) uplift **<+2-5%p** 시 원복 또는 재검토"와 같은 기준을 사용합니다.
    *   최소 query 수: 1인 사용자의 경우 통계적 유의성(p-value) 확보는 거의 불가능합니다. 대신, 추세 분석을 위해 확장된 키워드와 관련된 쿼리를 최소 **50-100건** 이상 수집 후 정성적/정량적 평가를 권장합니다.

*   **(c) 우리 상황과의 주요 차이**:
    1.  1인 사용자 환경은 일반적인 A/B 테스트 기반의 통계적 검증이 불가능하여, 사용자 피드백과 같은 정성적 평가의 중요성이 훨씬 큽니다.
    2.  'DA System'이라는 매우 특정한 도메인에서는 소수의 키워드 추가가 전체 hit rate에 큰 영향을 미칠 수 있습니다.
    3.  'hit rate'의 정의(e.g., top-k recall, user click)가 공개 벤치마크와 다를 수 있어 uplift 수치를 직접 비교하기 어렵습니다.
